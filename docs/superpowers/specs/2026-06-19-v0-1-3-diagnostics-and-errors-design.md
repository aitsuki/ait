# v0.1.3 诊断入口和错误提示设计

## 背景

`ait` v0.1.2 已经补齐开机自启和版本显示，开始更像一个可以日常常驻使用的小工具。上线后下一步最重要的问题不是继续扩功能，而是让用户遇到“翻译不了”“没反应”“报错”时，能更容易自查，也能更容易把有效信息反馈给维护者。

当前应用已经有文件日志，日志目录由 `logging::init_logging()` 创建，但用户没有入口打开日志目录。当前应用也已经有 `AppError::user_summary()`，但翻译窗口部分路径仍直接展示底层错误文本，用户可能看到过长或不够可操作的错误。

v0.1.3 的目标是用一个小版本补齐反馈链路：用户能打开日志，能复制诊断信息，也能看到更可读的常见错误提示。

## 目标

- 在托盘菜单增加 `打开日志目录`。
- 在设置窗口增加 `复制诊断信息`。
- 翻译窗口统一显示用户可读的错误摘要。
- 详细错误继续写入日志，便于维护者排查。
- 诊断信息不包含 API Key、原文、译文、完整配置内容等敏感信息。
- 保持 v0.1.3 范围小，优先解决发布后真实反馈阻力。

## 非目标

- 不做自动上传日志。
- 不做崩溃上报。
- 不做内置日志查看器。
- 不做自动更新或更新检查。
- 不做完整系统信息采集。
- 不改变翻译配置格式。
- 不新增联网诊断。

## 诊断信息内容

`复制诊断信息` 复制一段纯文本到剪贴板，格式以可读为主，不要求机器可解析。建议包含：

- ait 版本，例如 `ait v0.1.3`。
- 操作系统标识：如果当前代码能稳定取得 Windows 版本则包含，否则先写 `Windows`。
- 配置目录路径。
- 日志目录路径。
- 当前默认翻译配置名称和 provider 类型。
- 当前快捷键。
- 开机自启状态。

诊断信息必须排除：

- API Key 和加密后的 API Key。
- 原文和译文。
- 完整 `settings.json` 内容。
- 剪贴板内容。
- 用户本地文件内容。

如果某个字段读取失败，应在对应行写明读取失败的简短原因，而不是让整个复制动作失败。例如自启状态读取失败时，诊断信息里可以写：

```text
开机自启: 读取失败
```

## 托盘菜单行为

托盘菜单新增一项：

```text
打开日志目录
```

建议放在 `设置` 下面、分隔线上面。点击后调用 Windows Shell 打开日志目录。

日志目录沿用 `logging::init_logging()` 的目录规则：

```text
ProjectDirs::from("dev", "aitsu", "ait").data_local_dir()/logs
```

为了避免应用运行时只在 `run()` 里拿到日志目录但 UI 侧无法复用，应把日志目录解析封装成可复用函数，例如：

- `logging::log_dir() -> Result<PathBuf>`
- `logging::init_logging() -> Result<PathBuf>`

`init_logging()` 内部调用 `log_dir()`，托盘打开日志目录也调用 `log_dir()`。这样目录规则只有一个来源。

如果打开日志目录失败，显示简短消息框：

```text
无法打开日志目录，请稍后重试。
```

详细错误写日志。

## 设置窗口行为

设置窗口增加一个按钮：

```text
复制诊断信息
```

建议放在窗口底部版本号附近，作为辅助操作，不参与保存流程。

点击按钮时：

1. 收集诊断字段。
2. 复制到剪贴板。
3. 成功后显示：

```text
诊断信息已复制。
```

失败时显示：

```text
复制诊断信息失败，请打开日志目录后反馈日志文件。
```

详细错误写日志。

诊断信息不要求设置窗口先保存。它应基于当前已保存配置和系统真实状态生成，避免把用户正在编辑但尚未保存的 API 设置放进诊断内容。

## 错误提示行为

翻译窗口面向用户显示 `AppError::user_summary()`，不要直接显示底层错误的完整 `Display` 文本。详细错误继续通过 `tracing::warn!` 或现有日志路径记录。

重点覆盖这些常见场景：

- 取词失败：提示用户可以手动粘贴文本后重试。
- 网络失败：提示检查网络或代理。
- API Key 读取失败：提示重新保存接口配置。
- API Key 缺失或认证失败：提示检查 API Key。
- 翻译服务限流：提示稍后重试或切换提供方。
- 内置 Google 接口变化：提示重试或切换到 OpenAI 兼容接口。

`AppError::Translate(msg)` 里已经是面向用户的翻译错误时，可以继续显示 `翻译失败：{msg}`。如果未来引入更结构化的翻译错误类型，再把摘要逻辑下沉到错误类型本身。

## 模块边界

建议新增或扩展一个诊断边界，避免把路径、配置摘要、剪贴板逻辑散落在窗口代码里。

可选结构：

- `src/diagnostics.rs`
  - `DiagnosticInfo`
  - `DiagnosticInfo::collect(...) -> DiagnosticInfo`
  - `DiagnosticInfo::to_clipboard_text(&self) -> String`

设置窗口只负责触发复制和显示结果，不负责拼接诊断文本。

日志路径继续留在 `src/logging.rs`：

- `log_dir() -> Result<PathBuf>`
- `init_logging() -> Result<PathBuf>`
- `safe_text_len(text: &str) -> usize`

托盘菜单只负责产生 action。打开目录的具体动作放在 `app.rs` 或小的 Windows Shell 边界函数里，方便测试菜单 ID 到 action 的映射。

## 数据和隐私

v0.1.3 不新增持久化配置，不改变 `settings.json`。

诊断信息是用户主动点击后复制到剪贴板的文本，不自动发送到任何地方。文案和实现都应避免给用户“自动上传”的感觉。

日志中仍应避免记录原文、译文和 API Key。现有日志如果需要记录文本相关信息，应继续使用长度、provider、profile id 等低敏信息。

## 测试策略

优先测试不依赖真实 Windows UI 的纯逻辑：

- `logging::log_dir()` 和 `init_logging()` 使用同一目录规则。
- 诊断文本包含版本、配置目录、日志目录、快捷键、自启状态。
- 诊断文本不包含 API Key、加密 API Key、原文、译文。
- 托盘菜单新 ID 能映射到 `TrayAction::OpenLogDirectory`。
- 旧的已移除菜单 ID 行为按新设计更新，不再误判。
- 常见 `AppError` 的 `user_summary()` 输出稳定、可读。

Windows 边界通过小函数隔离：

- 打开日志目录成功时调用 Shell 打开路径。
- 打开失败时返回错误，由上层显示消息并写日志。
- 复制诊断信息失败时返回错误，由设置窗口显示失败消息。

实现后需要运行：

```powershell
cargo test
```

如果改到 Windows API feature，需要确认 `Cargo.toml` 的 `windows` features 覆盖 Shell 打开目录和剪贴板写入所需 API。
