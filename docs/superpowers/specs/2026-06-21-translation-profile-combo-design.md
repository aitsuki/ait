# 翻译窗口配置下拉框现代化设计

## 背景

按钮和输入框已经完成现代化，但翻译窗口顶部用于选择翻译配置的 `COMBOBOX` 仍使用 Win32 原生边框和背景。它位于窗口第一行，紧邻“有新版本”按钮和“原文”标签，是用户打开翻译窗口时最先看到的控件之一。

本阶段只处理翻译窗口顶部的配置下拉框，不处理设置窗口左侧配置列表。设置窗口当前使用的是 `LISTBOX`，不属于本阶段范围。

## 目标

- 让翻译窗口顶部配置下拉框的边框、背景和焦点态与现代化按钮、输入框保持一致。
- 保留 Win32 原生 `COMBOBOX` 的下拉、键盘选择、鼠标滚轮、焦点和列表行为。
- 只覆盖 `ID_PROFILE_COMBO`，避免影响其它潜在下拉框。
- 维持现有配置刷新、选中项读取和配置切换消息流程不变。
- 为后续其它选择类控件现代化沉淀独立的 `ui::combo` 模块。

## 非目标

- 不自绘下拉列表项。
- 不使用 `CBS_OWNERDRAWFIXED` 或自定义 ComboBox 窗口类。
- 不改变翻译配置数据结构、默认配置逻辑或切换流程。
- 不调整翻译窗口布局、控件位置或下拉高度。
- 不处理设置窗口 `LISTBOX`、复选框或其它控件。
- 不引入第三方 UI 框架。

## 覆盖范围

唯一覆盖控件：

- 翻译窗口顶部配置下拉框：`ID_PROFILE_COMBO = 2106`

该控件当前由 `src/ui/translate_window.rs` 中的 `create_combo` 创建，使用 `CBS_DROPDOWNLIST | WS_VSCROLL`，刷新逻辑走 `reset_combo_items`，选择变化继续通过 `CBN_SELCHANGE` 触发 `WM_TRANSLATE_WINDOW_PROFILE_CHANGED`。

## 视觉状态

普通：

- 白色背景。
- 深色文字。
- 浅灰圆角边框。

焦点：

- 白色背景。
- 深色文字。
- 蓝色圆角边框。

下拉打开：

- 与焦点态一致，边框保持蓝色，帮助用户识别当前活动控件。

禁用：

- 浅灰背景。
- 灰色文字。
- 浅灰边框。

颜色应复用现有按钮和输入框的克制工具型风格：普通边框使用浅蓝灰，活动边框使用安静蓝色，不使用大面积高饱和背景。

## 技术方案

保留 Win32 原生 `COMBOBOX`，采用“原生行为 + 外围现代化”的方式。

新增 `src/ui/combo.rs`：

- 定义 `ComboVisualState`，包含 `focused`、`dropped`、`disabled`。
- 定义 `ComboPalette`，包含背景、边框、文字颜色。
- 提供 `combo_palette(state)` 纯逻辑函数，便于单元测试。
- 提供 `combo_kind_for_control(id)` 或 `is_modern_combo(id)`，只识别 `ID_PROFILE_COMBO`。
- 提供 `combo_uses_native_border(id)`，现代化下拉框返回 `false`。
- Windows 下提供轻量子类安装函数，监听焦点、启用、下拉打开和关闭相关消息，并触发父窗口边框重绘。

创建控件时：

- `create_combo` 继续创建 `COMBOBOX`。
- 保留 `CBS_DROPDOWNLIST | WS_VSCROLL`。
- 对现代化下拉框不再附加 `WS_BORDER`。
- 创建后安装 ComboBox 子类，用于焦点和下拉状态重绘。

父窗口职责：

- 在 `WM_PAINT` 中调用统一辅助函数绘制 `ID_PROFILE_COMBO` 外围 1px 圆角边框。
- 在需要处理 ComboBox 颜色的消息中设置背景和文字颜色；如果 Win32 对下拉框主体颜色支持有限，第一版以外围边框现代化为主，不接管列表项绘制。
- 未登记控件保持默认处理。

边框绘制方式应贴近 `ui::edit`：由父窗口根据子控件矩形映射到父窗口坐标后绘制，不覆盖下拉按钮区域，不影响系统绘制的箭头、选中文本和下拉列表。

## 消息和状态

ComboBox 子类需要关注：

- `WM_SETFOCUS`：进入焦点态并重绘边框。
- `WM_KILLFOCUS`：离开焦点态并重绘边框。
- `WM_ENABLE`：启用状态变化后重绘边框。
- `WM_NCDESTROY`：移除子类，避免悬挂回调。

配置切换仍由父窗口现有 `WM_COMMAND` 分支处理：

- `ID_PROFILE_COMBO`
- `CBN_DROPDOWN` 和 `CBN_CLOSEUP`：同步下拉打开状态并重绘边框。
- `CBN_SELCHANGE`
- `WM_TRANSLATE_WINDOW_PROFILE_CHANGED`

现代化样式不得改变这些消息的发送时机和语义。

## 错误处理

创建 ComboBox 失败沿用现有 `AppError::Windows`。

安装子类失败返回 `AppError::Windows("安装下拉框焦点处理失败")` 或同等明确错误。

绘制阶段不返回业务错误。GDI 对象创建失败时应跳过边框绘制或退回默认外观，不能影响下拉框交互和翻译流程。

## 测试

自动化测试覆盖纯逻辑：

- `ID_PROFILE_COMBO` 能映射为现代化下拉框。
- 未知控件 id 不进入现代化下拉框路径。
- 现代化下拉框不使用原生 `WS_BORDER`。
- 普通、焦点、下拉打开、禁用状态对应颜色稳定。
- 焦点态和下拉打开态使用同一活动边框色。
- 边框绘制矩形不会缩小控件可交互区域。

现有测试继续覆盖：

- 翻译窗口布局。
- 配置列表生成和选中项。
- 配置切换消息处理。
- 按钮和输入框现代化逻辑。

手动验证：

- 打开翻译窗口，确认顶部配置下拉框边框和输入框风格一致。
- 使用鼠标点击下拉框，确认焦点边框清晰。
- 展开下拉列表，确认列表正常显示，选项可点击。
- 使用键盘 Tab 到下拉框，确认焦点态可见。
- 使用方向键切换配置，确认配置变更仍生效。
- 窗口 resize 后，下拉框边框位置正确，没有残影。
- 有新版本按钮显示或隐藏时，下拉框布局不受影响。

## 风险

- Win32 `COMBOBOX` 的主体、下拉按钮和弹出列表由系统分段绘制，第一版不应为了完全统一外观而接管列表项绘制。
- 去掉 `WS_BORDER` 后，需要父窗口稳定绘制外围边框，否则 resize 或焦点切换时可能出现短暂旧背景。
- 下拉打开状态不一定只通过一个消息路径暴露，需要实现时验证实际消息顺序。
- ComboBox 弹出的列表窗口不是普通子控件，本阶段不修改其边框和列表项样式，避免引入不可控行为。

## 验收

- 翻译窗口顶部配置下拉框使用现代化圆角边框。
- 普通、焦点、下拉打开和禁用状态视觉明确。
- 配置选项刷新、选中项读取和配置切换行为保持不变。
- 下拉列表、键盘导航和鼠标选择仍使用原生行为并正常工作。
- 所有现有测试继续通过。
- 新增测试覆盖 ComboBox 视觉状态、控件映射和原生边框策略。
