# 现代 Win32 UI 字体设计

## 背景

当前翻译窗口和设置窗口都直接创建原生 Win32 控件，但没有显式设置控件字体。未设置字体时，控件可能使用较老的默认 GUI 字体，在中文文本和高 DPI 环境下看起来偏糊、不够现代。

## 目标

- 将应用内 Win32 控件统一切换到更清晰、现代的系统 UI 字体。
- 覆盖翻译窗口和设置窗口中的标签、输入框、按钮、列表框、下拉框等控件。
- 不引入外部字体文件，不增加安装包体积。
- 保持现有窗口布局、尺寸、交互和业务逻辑不变。

## 方案

新增一个 Windows-only 的 UI 字体助手模块，负责创建并复用一个全局 `HFONT`。字体使用 `Microsoft YaHei UI`，字号为 9pt。该字体是 Windows 上常见的中文 UI 字体，适合混排中文和英文，也比未显式指定的老默认字体更清晰。

控件创建成功后，统一发送 `WM_SETFONT`，并要求控件立即重绘。这样每个控件都使用同一套字体，不需要在每个调用点手动重复 GDI 细节。

## 组件

- `src/ui/font.rs`
  - 提供 Windows-only 字体创建和应用函数。
  - 使用 `OnceLock` 复用字体句柄，避免重复创建字体资源。
  - 根据当前设备 DPI 将 9pt 转换为逻辑像素高度。

- `src/ui/mod.rs`
  - 暴露字体模块给具体窗口使用。

- `src/ui/translate_window.rs`
  - 在 `create_control` 创建控件后应用统一字体。

- `src/ui/settings_window.rs`
  - 在 `create_control` 创建控件后应用统一字体。

## 错误处理

如果字体创建失败，控件创建流程返回现有的 `AppError::Windows` 错误。这样可以避免悄悄退回模糊字体，也能在日志或调用方看到明确失败原因。

## 测试

- 增加纯逻辑测试，验证 9pt 到字体高度的换算规则为负值逻辑高度。
- 运行现有测试，确认字体改动没有影响配置、窗口布局、翻译流程等逻辑。
- 在 Windows 环境可手动打开翻译窗口和设置窗口，确认控件字体变清晰且没有文本被截断。

## 非目标

- 不更换 UI 框架。
- 不重做窗口布局。
- 不打包第三方字体。
- 不调整颜色、边框、间距或控件样式。
