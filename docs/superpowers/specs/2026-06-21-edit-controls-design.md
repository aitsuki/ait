# Win32 输入框现代化设计

## 背景

按钮已经通过 owner-draw 完成现代化，但设置窗口和翻译窗口中的 `EDIT` 控件仍使用原生旧边框。按钮变新后，输入框、特别是翻译窗口的原文和译文多行框会显得不协调。

本阶段只处理 `EDIT` 控件。翻译窗口的原文和译文是多行输入框，承担滚动、选择、复制、中文输入、换行显示和只读展示等行为，因此不能接管文本区域绘制。

## 目标

- 统一设置窗口单行输入框和翻译窗口多行输入框的视觉风格。
- 保留 Win32 原生 `EDIT` 的文本渲染、光标、选择、滚动条、IME、复制粘贴和键盘行为。
- 为普通、焦点、只读、禁用状态提供稳定的背景、文字和边框颜色。
- 保持现有窗口布局、控件尺寸、业务逻辑和消息流程不变。
- 为后续 `COMBOBOX`、`LISTBOX` 等控件现代化沉淀可复用的样式模块思路。

## 非目标

- 不自绘多行文本内容。
- 不自定义滚动条。
- 不重写 `EDIT` 控件类。
- 不引入第三方 UI 框架。
- 不调整窗口布局、控件位置或尺寸。
- 不处理按钮、下拉框、列表框、复选框或窗口背景。
- 不改变翻译、设置保存、API Key 显示隐藏、快捷键捕获等业务行为。

## 覆盖范围

设置窗口单行输入框：

- 快捷键
- 名称
- Base URL
- 模型
- API Key
- 超时秒数

翻译窗口多行输入框：

- 原文
- 译文

译文框当前是只读 `EDIT`，视觉上应表现为只读内容区，但仍允许选择和复制。

## 视觉状态

普通可编辑：

- 白色背景。
- 深色文字。
- 浅灰边框。

焦点：

- 白色背景。
- 深色文字。
- 蓝色边框。
- 不额外绘制会干扰文本选择的内部焦点框。

只读：

- 极浅灰背景。
- 深色文字。
- 浅灰边框。
- 用于译文框和其它只读输入场景。

禁用：

- 浅灰背景。
- 灰色文字。
- 浅灰边框。

颜色应与现有现代按钮保持克制、工具型应用的风格，不使用大面积高饱和背景。

## 技术方案

保留 Win32 原生 `EDIT` 控件，采用“原生内容 + 外围现代化”的方式。

新增 `src/ui/edit.rs`：

- 定义 `EditVisualState`，包含 `focused`、`readonly`、`disabled` 等状态。
- 定义 `EditPalette`，包含背景、边框、文字颜色。
- 提供 `edit_palette(state)` 等纯逻辑函数，便于单元测试。
- 提供控件 id 到现代化输入框类型的映射。
- Windows 下提供轻量子类安装函数，用于处理 `WM_SETFOCUS`、`WM_KILLFOCUS`、`WM_ENABLE`、`WM_NCDESTROY` 等消息并触发父窗口重绘。

父窗口职责：

- 创建现代化 `EDIT` 时不依赖原生 `WS_BORDER` 的旧式边框。
- 在 `WM_CTLCOLOREDIT` 中为可编辑输入框设置文本色、背景色并返回背景刷。
- 在 `WM_CTLCOLORSTATIC` 中为只读或禁用输入框设置文本色、背景色并返回背景刷。
- 在需要重绘时绘制 `EDIT` 外围 1px 边框。
- 未登记的控件保持现有默认处理。

边框绘制可以放在父窗口 `WM_PAINT` 或一个统一的辅助函数中，由窗口在已有绘制流程里调用。边框只画在控件外侧或控件矩形边缘，不覆盖文本内容，也不覆盖滚动条区域。

## 翻译窗口多行输入框约束

翻译窗口已有多行编辑框子类，用于：

- `Ctrl+A` 全选。
- `Escape` 隐藏窗口。
- 三击选择当前段落。

输入框现代化不能破坏这些行为。实现时可以选择：

- 在现有多行编辑框子类中追加焦点重绘逻辑；或
- 安装独立子类，但必须使用不同 subclass id，并保证 `DefSubclassProc` 链路正常。

优先选择最小改动方案：复用或扩展已有编辑框子类，避免多个子类之间的生命周期和状态管理复杂化。

多行文本区域继续由系统绘制，确保：

- 滚动条可用。
- 文本选择高亮正常。
- 中文输入和 IME 候选窗口正常。
- 只读译文可选择、复制。
- 换行规范化逻辑不变。

## 设置窗口单行输入框约束

设置窗口单行输入框需要覆盖：

- 常规文本输入。
- API Key 密码模式。
- API Key 显示/隐藏切换。
- 快捷键捕获只读输入框。
- Google 配置隐藏网络字段后的背景擦除。

现代化样式只改变视觉，不改变读取、写入、密码字符、只读、显示隐藏、启用禁用等行为。

## 错误处理

创建或子类安装失败沿用现有 `AppError::Windows`。

颜色刷或 GDI 对象创建失败时，绘制逻辑应退回默认处理或跳过外围边框，不能影响输入框文本交互。

未登记控件不应被现代化输入框处理，避免误伤其它控件。

## 测试

自动化测试覆盖纯逻辑和接入判断：

- 已知设置窗口单行输入框 id 能映射为现代化输入框。
- 已知翻译窗口多行输入框 id 能映射为现代化输入框。
- 未知控件 id 不进入现代化输入框路径。
- 普通、焦点、只读、禁用状态对应的背景、边框、文字颜色稳定。
- 只读状态和禁用状态颜色不同，便于区分可复制内容和不可用控件。
- 现有多行输入框行为测试继续通过，包括换行规范化、`Ctrl+A`、`Escape` 和三击选段。

手动验证：

- 打开设置窗口，确认所有单行输入框边框统一且焦点清晰。
- API Key 在显示和隐藏之间切换后，背景、文字、边框正常。
- 选择 Google 配置时，隐藏的网络字段区域没有残影。
- 打开翻译窗口，确认原文和译文多行框视觉统一。
- 原文框可输入中文、选择、复制、滚动。
- 译文框只读但可选择、复制、滚动。
- 焦点在原文、译文、按钮和下拉框之间切换时没有明显闪烁或边框残留。

## 风险

- 多行 `EDIT` 对滚动条和文本区域很敏感，边框绘制不能覆盖滚动条或内容。
- `WM_CTLCOLOREDIT` 和 `WM_CTLCOLORSTATIC` 对可编辑、只读、禁用控件的触发路径不同，需要分别处理。
- 翻译窗口已有编辑框子类，新增焦点重绘逻辑时必须保持原有快捷键和三击选择行为。
- 去掉 `WS_BORDER` 后需要确认控件内容不会贴边过近；如有必要，可只在后续小阶段考虑 `EM_SETMARGINS`，本阶段不主动改变文本边距。

## 验收

- 设置窗口单行输入框和翻译窗口多行输入框使用统一现代边框和颜色。
- 翻译窗口多行原文和译文仍保持现有输入、滚动、选择、复制行为。
- 译文只读状态视觉明确，但不阻止选择复制。
- 所有现有测试继续通过。
- 新增测试覆盖输入框视觉状态和控件映射。
