# v0.2.0 更新与发布透明度设计

## 背景

`ait` 现在已经具备可日常使用的核心翻译能力，也有自动发布 GitHub Release 的基础设施。下一步最需要补的是“用户怎么知道有新版本”以及“用户怎么确认下载来源是官方发布”。

当前仓库已经有：

- `Release` GitHub Actions 工作流。
- Inno Setup 安装器脚本。
- `README.md` 中的下载说明。
- 设置窗口与托盘入口。

但还缺少一条完整的更新链路：

- 应用内无法检查 GitHub Releases 是否有新版。
- 用户没有一个稳定的“打开最新版 Release 页”的入口。
- 发布页和 README 里对安装包来源、校验信息、未签名状态的说明还不够完整。
- 安装器体验还可以再收紧，尤其是版本信息和安装后行为的一致性。

v0.2.0 的目标就是把这条链路补完整，让用户更容易拿到新版本，也更容易确认自己下载的是官方构建。

## 目标

- 启动时自动检查一次更新，但不打断启动流程。
- 在设置页提供手动“检查更新”入口。
- 检查到新版后，明确告诉用户当前版本与最新版本，并提供打开 GitHub Releases 最新页的操作。
- 提供一个稳定的“打开最新版 Release 页面”入口。
- 发布说明中提供安装包和便携版的校验信息。
- `README.md` 和常见问题明确说明当前未做代码签名，以及如何确认下载来源。
- 让安装器的版本信息、安装后启动行为和卸载入口保持清晰一致。

## 非目标

- 不做自动下载。
- 不做自动安装更新。
- 不做后台静默升级。
- 不做代码签名。
- 不做多平台发布。
- 不做 zip 分发包。
- 不改翻译配置格式。
- 不改核心翻译流程。

## 体验原则

- 更新检查必须是低打扰的，不应阻塞启动或翻译流程。
- 用户可以忽略更新提示，不会影响继续使用。
- 所有“下载最新版”的路径都应指向 GitHub 官方 Release。
- 发布透明度优先于复杂自动化，不引入额外凭证或私有分发逻辑。

## 更新检查行为

更新检查分为两条入口：

1. 启动时自动检查一次。
2. 设置页手动点击“检查更新”。

两条入口都调用同一套逻辑，避免版本比较、网络请求和提示文案分叉。

检查结果至少分三类：

- 已是最新版本。
- 发现新版可用。
- 检查失败。

发现新版时，界面应展示：

- 当前版本。
- 最新版本。
- 一个打开 GitHub Releases 最新页的操作。

检查失败时，只给出简短提示，不阻断正常使用。详细原因写入日志。

## 最新版本来源

最新版判断以 GitHub Releases 最新正式版为准，优先使用仓库自己的官方发布页，而不是第三方镜像。

推荐的来源路径是：

- `https://github.com/aitsuki/ait/releases/latest`

如果后续需要更细的版本判断逻辑，可以在应用内部增加一个小的更新模块，负责：

- 读取当前 `CARGO_PKG_VERSION`。
- 获取 GitHub Releases 最新版本信息。
- 比较版本号。
- 生成提示状态。

## 发布透明度

Release 说明中应补齐以下信息：

- 安装包文件名。
- 便携版文件名。
- 校验信息，例如 SHA256。
- 下载来源说明。

README 与 FAQ 应明确说明：

- 当前没有代码签名。
- 发行文件应从 GitHub Releases 下载。
- 用户可以通过 Release 页面标题、附件名和校验信息确认来源。

这里的目的不是让用户自己做复杂审计，而是把官方来源说清楚，把容易误解的地方先讲明白。

## 安装器体验

安装器仍然使用 Inno Setup，不引入新的打包工具。

安装器应保持以下行为：

- 安装到用户程序目录。
- 提供安装后启动。
- 保留标准卸载入口。
- 显示清晰的版本信息。
- 和 Release 版本号保持一致。

如果现有脚本已经满足部分行为，则只补剩余缺口，不做无意义重写。

## 模块边界

建议新增一个轻量更新边界，避免把网络请求和版本比较直接散落在 UI 里。

可选结构：

- `src/update.rs`
  - `UpdateCheckStatus`
  - `UpdateInfo`
  - `check_for_update(...)`
  - `latest_release_url()`

UI 层只负责触发检查、展示状态、打开链接，不负责拼接 GitHub URL 或解析版本号。

发布说明和 README 仍由现有文档与工作流负责：

- `.github/workflows/release.yml`
- `README.md`

如果 Release 校验信息需要自动生成，逻辑应放在工作流里，不混进 Rust 运行时。

## 数据与错误处理

更新检查的网络错误应视为可恢复错误：

- 不影响启动。
- 不影响翻译。
- 不影响设置保存。

用户侧只看到简短结果，例如：

- “暂时无法检查更新”
- “已是最新版本”
- “发现新版本 vX.Y.Z”

详细错误继续写日志，供维护者排查。

## 测试策略

优先测试不依赖真实网络和真实 GitHub 的纯逻辑：

- 版本号比较逻辑。
- GitHub Release JSON 解析。
- 更新状态到 UI 文案的映射。
- “打开最新版 Release 页面”的链接生成。
- Release 说明文本包含安装包、便携版和校验信息。
- README 中包含未签名和来源说明。

工作流层面验证：

- `cargo test` 通过。
- `cargo build --release` 通过。
- GitHub Actions 的 `Release` workflow 仍能生成两个附件。
- 发布说明里能看见校验信息。

## 发布说明

v0.2.0 发布说明应包含：

- 启动时检查更新。
- 设置页手动检查更新。
- 可从应用内打开 GitHub Releases 最新页。
- Release 附件提供校验信息。
- README 补充官方来源和未签名说明。
- 安装器体验有所收紧。
